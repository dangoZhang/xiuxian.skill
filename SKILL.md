# vibecoding.skill

## 这是什么

`vibecoding.skill` 读取真实的 Code Agent 记录，默认做四件事：

- 判断当前 vibecoding 等级
- 生成一份可读的协作画像
- 生成一张可分享的等级卡
- 给出下一轮最该补的动作

用户明确要求时，再导出共享包，让另一个也在用这个仓库的人复用这套协作方式。

这里要区分两层：

- `vibecoding.skill`：蒸馏项目，也是入口 skill。负责读记录、做判断、生成卡片、导出结果。
- `xxx.skill`：蒸馏结果 skill。负责复用某个人已经蒸出来的协作方式。

也要区分两类画像：

- 直接画像：基于原始轨迹做等级、阶段、强项、短板判断。这是主判断。
- 间接画像：基于长历史蒸馏出风格标签、摘要页和共享包画像。它必须先对齐直接画像，再补充风格细节。

## 默认能力

默认回答优先给这些内容：

1. 当前等级和阶段
2. 习惯画像
3. 最稳的地方
4. 最该补的地方

用户继续追问时，再补：

- 分享卡
- 完整报告
- 共享包
- 目标等级训练建议

## 适用场景

当用户想做下面这些事时，就该调用这个 skill：

- “帮我看看我最近的 vibecoding 等级”
- “给我一个我和 AI 协作的画像”
- “给我做一张可以分享的等级卡”
- “告诉我下一轮先补什么”
- “把我这套协作方式导出来，给另一个也装了这个仓库的人继续用”
- “按这份画像继续和我协作”

## 工作流

1. 先判断输入是单次会话、时间窗聚合，还是导出包。
2. 自动读取可用轨迹。
3. 轨迹过长时，保留用户 prompt 原文，AI 回复只做取头压缩，再按用户回合分块。
4. 分块时不要伪造 chunk 级 tool call、token 和验证证据；这类统计只在总报告层回收原始值。
5. 先做直接画像，输出等级、阶段、强项、短板。
6. 再做间接画像，但间接画像要复用直接画像的主结论，只把 16 维蒸馏当成标签、摘要和共享包补充层。
7. 用户明确要求“按最新术语”“结合最新 agent 进展”“联网更新词汇”时，先刷新官方术语输入，再写报告。
8. 先输出人话版结论：
   - 当前等级
   - 协作画像
   - 强项
   - 短板
9. 用户要分享卡时，再生成卡片。
10. 用户要共享时，再导出完整共享包。
11. 用户要升级时，再给训练方案。

联网刷新术语时，优先参考这些官方来源：

- OpenAI Codex / model docs
- Anthropic Claude Code docs
- Cursor docs
- MCP 官方文档

刷新后会生成三份输入文件：

- `docs/latest-agent-terms.md`
- `docs/latest-agent-terms.json`
- `docs/latest-agent-terms.prompt.md`

## 结果 skill 接管

当用户给出某个蒸馏结果 skill，或给出导出包并要求“按他的方式继续协作”时：

1. 先把 `vibecoding.skill` 当入口 skill。
2. 读取导出包里的 `PROFILE.md`、`REPORT.md`、`SKILL.md`。
3. 识别其中的结果 skill，例如 `同事.skill`。
4. 后续协作默认按这个结果 skill 的行为层执行。

可以理解成：

- `vibecoding.skill` 负责蒸馏和路由。
- 结果 skill 负责真正复用某个人的协作方式。

当前实现里，一个导出包只放一个二级结果 skill，当前会话也只接管一个结果 skill。

## 导出示例话术

用户可以直接通过和 Agent 的自然语言对话触发导出，不需要先手动跑 CLI。

下面这些是固定示例话术，用来说明怎么说最稳，不把它们当成通用触发词规则：

- “把我这套协作方式导出来”
- “导出成一个共享 skill”
- “给我一份能发给同事继续用的导出包”
- “把我最近这段记录整理成一个别人能接着用的包”

触发后默认动作：

1. 读取指定记录或最近时间窗。
2. 生成画像、报告、分享卡。
3. 导出完整共享包。
4. 明确告诉用户导出了什么，以及接收方下一句该怎么说。

## 共享与模仿

当用户说“把我这套能力分享出去”或“按他的方式和我协作”时：

- 把导出包理解成协作方式的共享包。
- 更合适的场景是双方都已经在自己的宿主里装了 `vibecoding.skill`。
- 接收方先读 `PROFILE.md` 和 `REPORT.md`，再按里面的节奏协作。
- 不要把它写成人格扮演。

导出时默认带上：

- `SKILL.md`
- `PROFILE.md`
- `REPORT.md`
- `snapshot.json`
- `assets/vibecoding-card.png`

## 语言规则

- 默认先说人话。
- 用户提到“修仙”或“境界”时，分享卡、画像标题和相关文案切到修仙风格。
- 修仙风格下，优先用“境界、神识、傀儡、宗门、法宝”这类词；关键 AI 术语放在括号里保留原意。
- 用户明确要求英文，或轨迹几乎全是英文时，用英文输出。
- 文案要短，能直接复制给别人看。

## 好的用法

- “帮我看下我最近 14 天的 vibecoding 等级，顺便总结我和 AI 的协作习惯。”
- “给我一个我最近这段时间的 vibe coding 水平画像。”
- “给我生成一张最近两周的 vibecoding 等级卡，文案要适合直接分享。”
- “如果我想把这套协作方式继续打磨，下一轮先练什么？”
- “把我最近这段协作记录蒸馏成一个共享 skill，给另一个也装了这个仓库的人用。”
- “把我最近两周的协作方式直接导出来，给我共享包和接收方该怎么用的一句话。”
- “这是同事的导出包。先用 vibecoding.skill 读画像，再调用里面的结果 skill 和我协作。”
- “这是同事的导出包。先读他的画像，再按这套方式和我一起做当前任务。”

## 输出约定

默认先给：

1. 一段总览
2. 等级
3. 一段画像判断
4. 一条下一步建议

用户继续追问时，再给更长的拆解、卡片、共享包和升级计划。
